# 1. TypeScriptとは何か

## What

TypeScriptは、JavaScriptに<u>型システム</u>を加えた言語です。

### 型って何？

型は<u>その値がどんな種類のデータか</u>を表す情報です。
たとえば、`"Akane"` は文字列（`string`）、`29` は数値（`number`）です。

型の効果

- コードを読む人が<u>この変数には何が入るのか</u>をすぐ理解できます。
- エディタも<u>この変数には何が入るのか</u>を認識してくれます。
- 意図しない値を代入するとエラーが出ます。

```ts
const userName: string = "Akane";
const age: number = 29;
```

### 静的型と動的型

JavaScriptは動的型付け言語です。実行してはじめて型の問題が見えることがあります。

```js
let user: User = { name: "Akane" };
...
user = null;
...
console.log(user.name); // user は null！ 実行時エラー！
```

TypeScriptは静的型チェックを行うため、実行前に問題を検出できます。

```ts
let user: User = { name: "Akane" };
...
user = null; // そもそも user に null を代入できない
...
console.log(user.name);
```

- 動的型付け: 実行時に型（データ表現）が確定・検証される
- 静的型付け: 開発時（ビルド時）に型（取りうる値）の整合性を確認する

そもそも、動的型と静的型は<u>全くの別物</u>です！

- 動的型: データ表現に関する情報を<u>実行時に</u>どこかに持っておく
- 静的型: 取りうる値を<u>開発時に</u>持っているだけで、実行時には存在しない！

TypeScriptは「実行時に処理される」のではなく、「コードを書いている最中に存在し、安全性を高める」と考えましょう。

### トランスパイルとは

トランスパイルは、TypeScriptコードをJavaScriptコードへ変換する処理です。
特にフロントエンドでは、ブラウザがTypeScriptを直接実行できないため、この変換が必要です。

流れとしては次のようになります。

1. `.ts` を書く
2. `tsc` や `vite` が `.js` へ変換する
3. 変換後の `.js` が実行される

つまり、最終的に動くのはJavaScriptです。

## Why

TypeScriptで「型」を使う主な理由は、バグを「実行前」に見つけるためです。

### 型によるバグの早期発見

たとえば、<u>数値にしか存在しない処理</u>と<u>文字列にしか存在しない処理</u>を取り違える、といったミスは普通に起こりえます。

JavaScriptでは、実行されるまでエラーが出ません。
実行時のエラーは「ユーザーの画面で発生します」！
つまり、アウトです。

TypeScriptなら実行前にエラーとして検出できます。

```ts
const total: number = 100;
total.toUpperCase(); // number 型にそんなメソッドはありません！
```

コードが複雑になるほど、基本的なミスが積み重なると発見が難しくなります。
しかし、自動で不整合に気づければ助かりますよね？

### 動的型付けではできないの？

基本的にはできません。
バグの見つけ方はこのような感じです。

- JavaScript: テストや実行時に問題を見つける
- TypeScript: それに加え、編集中や保存時、ビルド時に問題を見つける

動的型付けは、規模が大きくなるほど<u>壊れたことに気づくのが遅い</u>という問題が出やすくなります。
TypeScriptはこの遅れを減らすための道具です。

### 静的型の恩恵

静的型の恩恵はエラー検出だけではありません。

- 補完が強くなる（プロパティ候補、引数情報）
- 関数の仕様が型として残る（型がバグの無いドキュメントになる）
- リファクタ時に影響範囲を追いやすい
- チームでの読み書きが安全になりやすい

結果として、「動くコード」だけでなく「変更し続けられるコード」を作りやすくなります。

## How

最小の流れは、次の3ステップです。

1. `.ts` ファイルを書く
2. `tsc` で型チェックや変換を行う
3. 出力された `.js` を実行する

### tsc とは

`tsc` はTypeScript Compilerの略で、TypeScript公式のコンパイラです。
主な役割は次の2つです。

- 型チェックを行う
- 必要に応じてJavaScriptを出力する

`tsconfig.json` を通して、型チェックの厳しさ（`strict` など）も制御できます。

### 型チェックのみを実行する

```bash
npm exec tsc --noEmit
# pnpm の場合
pnpm exec tsc --noEmit
```

`--noEmit` を付けると、JavaScriptファイルを生成せずに型チェックだけ行えます。
学習中やCIでの品質チェックに便利です。

### vite が自動でやってくれるけど…

Viteを使っている場合、開発サーバー起動中にトランスパイルは自動で行われます。
ただし、プロジェクト全体を厳密に検証するには `tsc --noEmit` を別で実行するのが安全です。

- 開発中: Viteの変換で素早く確認
- 品質確認: `tsc --noEmit` で全体の型チェック

この2つを併用すると、開発速度と安全性を両立しやすくなります。

### コマンドを打つのが面倒

そんなときは、`package.json` に自作スクリプトを書きましょう！

```json
{
  ...
  "scripts": {
    "dev": "vite dev",
    "build": "vite build",
    "typecheck": "tsc --noEmit" // これを追加
  },
  ...
}
```

すると、以下で実行できます

```bash
npm run typecheck
# pnpm の場合
pnpm typecheck
```

## このページのまとめ

- TypeScriptはJavaScriptに型システムを加えた言語
- 型は「データの種類」を表し、意図しない値の混入を防ぐ
- `tsc` が型チェックと変換を担う
- `tsc --noEmit` は最終確認で特に有効
